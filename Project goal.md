#  Frontend Design Core
Universal Frontend Design & Clean Code Rules

---

## 1. فكرة النظام
NEXA Frontend Design Core هو نظام قواعد عام يحدد طريقة التفكير والتنفيذ في تصميم وبرمجة واجهات المستخدم Frontend، بغض النظر عن نوع المشروع أو مجاله.

هذا النظام لا يفرض ألوان، خطوط، أو Components جاهزة،  
بل يضع إطارًا فكريًا وتنفيذيًا يضمن الوضوح، الجودة، وقابلية التوسع والصيانة.

---

## 2. الهدف الأساسي
يهدف هذا النظام إلى:
- منع العشوائية في التصميم والتنفيذ
- توحيد أسلوب العمل داخل المشروع
- ضمان تجربة مستخدم واضحة وسلسة
- تقليل إعادة التصميم
- تسهيل التطوير والصيانة
- ضبط دور الذكاء الاصطناعي كمساعد لا كمقرر

---

## 3. نطاق النظام (Scope)

### يشمل:
- بنية الصفحات
- ترتيب المحتوى
- مبادئ UX الأساسية
- التدرج البصري (Visual Hierarchy)
- التفاعل (Interaction)
- التصميم المتجاوب (Responsive Design)
- الأداء
- نظافة الكود وقابليته للفهم والصيانة

### لا يشمل:
- اختيار ألوان محددة
- اختيار خطوط
- تصميم UI نهائي
- Components جاهزة
- أي منطق خاص بمجال معين

---

## 4. فلسفة التصميم الأساسية

### 4.1 المستخدم قبل الجمال
- أي عنصر لا يخدم المستخدم → لا مكان له
- الجمال نتيجة وليس هدفًا
- الوضوح أهم من الإبهار البصري

---

### 4.2 كل صفحة لها هدف واحد
- كل صفحة يجب أن تجيب على سؤال واحد:
  > ماذا نريد من المستخدم أن يفعل؟
- لا صفحات متعددة الأهداف
- لا تشتيت بصري أو وظيفي

---

### 4.3 لا مفاجآت
- المستخدم يجب أن يتوقع:
  - أين يضغط
  - ماذا سيحدث
  - إلى أين سينتقل

أي سلوك غير متوقع = تجربة سيئة

---

## 5. القواعد العامة للتصميم

### 5.1 التدرج البصري (Visual Hierarchy)
- العنوان الرئيسي هو أوضح عنصر
- العناوين الفرعية أقل بروزًا
- النصوص مريحة للقراءة
- الأزرار الأساسية واضحة
- الأزرار الثانوية أقل بروزًا

---

### 5.2 المساحات (Spacing)
- الفراغ عنصر تصميم أساسي
- لا تزاحم بين العناصر
- كل Section له مساحة تنفس

قاعدة:
> إذا احتجت Border للفصل، غالبًا المساحة خاطئة

---

### 5.3 الاتساق (Consistency)
- نفس العنصر = نفس السلوك
- نفس اللون = نفس المعنى
- نفس Layout = نفس الفكرة

---

### 5.4 الوضوح قبل الإبداع
- لا Animations إلا إذا خدمت الفهم
- لا Effects إلا إذا دعمت التجربة
- لا إبداع على حساب الوضوح

---

## 6. بنية الصفحة (Page Structure)
أي صفحة يجب أن تمر بالمراحل التالية:
1. Orientation — المستخدم يفهم أين هو
2. Context — المستخدم يفهم الصفحة عن ماذا
3. Content — عرض المعلومات الأساسية
4. Action — ماذا يجب أن يفعل المستخدم
5. Feedback — ماذا حدث بعد التفاعل

---

## 7. قواعد التفاعل (Interaction Rules)
- أي عنصر قابل للنقر يجب أن يكون واضحًا
- حالات Hover / Active / Focus إلزامية
- Feedback فوري (Loading / Success / Error)

---

## 8. التصميم المتجاوب (Responsive Design)
- التصميم يتكيف ولا ينكمش
- Mobile ليس Desktop مصغر
- الأولوية للمحتوى الأهم
- كل Breakpoint له تفكير خاص

---

## 9. الأداء جزء من التصميم
- التصميم الذي يضر الأداء = تصميم فاشل
- الصور محسوبة
- الأنيميشن محسوبة
- لا تأثير بصري بدون مبرر

---

## 10. ملكية التصميم (Design Ownership)
- تم تنفيذ تصميم UI/UX كامل مسبقًا قبل بدء البرمجة
- التصميم:
  - مدروس
  - بصريًا جذاب
  - سهل الاستخدام
  - مبني على تجربة مستخدم واضحة

⚠️ لذلك:
> أي كود Frontend يجب أن يلتزم حرفيًا بالتصميم  
> ولا يُسمح بإعادة تفسير التصميم بصريًا أو وظيفيًا

---

## 11. مصدر الحقيقة الوحيد (Single Source of Truth)
- صور التصميم (Screens) المرسلة لكل صفحة:
  - هي المرجع الوحيد
  - هي القرار النهائي
  - لا يتم تعديلها إلا بطلب صريح

- الكود:
  - يترجم التصميم
  - لا يعيد تصميمه

Design → Code  
وليس Code → Design

---

## 12. دور الذكاء الاصطناعي أو المطوّر

### مسموح له:
- تحويل التصميم إلى كود
- تحسين بنية الكود
- تنظيم الملفات
- اقتراح تحسينات تقنية غير مرئية للمستخدم
- تحسين الأداء
- تحسين قابلية الصيانة

### غير مسموح له:
- تغيير Layout
- تغيير الألوان
- تغيير المسافات
- إضافة عناصر جديدة
- حذف عناصر موجودة
- تغيير تجربة المستخدم

---

## 13. قواعد كتابة الكود (Clean Code Rules)

### 13.1 الوضوح قبل الذكاء
- الكود يجب أن:
  - يُقرأ بسهولة
  - يُفهم بدون شرح
  - لا يعتمد على حيل معقدة

الكود يُكتب للبشر أولًا، ثم للآلة.

---

### 13.2 فصل المسؤوليات (Frontend SoC)
- UI Components → شكل فقط
- Logic → منفصل
- Styles → منظمة
- State → واضح المصدر

لا:
- منطق عشوائي داخل JSX
- Styles متداخلة بدون نظام

---

### 13.3 قابلية الصيانة
أي Component يجب أن:
- يكون صغيرًا وواضحًا
- له اسم يعبر عن وظيفته
- لا يعتمد على سياق خفي

---

### 13.4 إعادة الاستخدام بدون تعقيد
- لا تعميم زائد
- لا Components مرنة أكثر من اللازم
- كل Component يخدم هدفًا واضحًا

---

## 14. بنية المشروع (Project Structure Rules)
- هيكل بسيط ومتوقع
- سهل التنقل
- أي شخص جديد:
  - يفهم أين يضع الكود
  - يفهم أين يجد ما يبحث عنه

---

## 15. الالتزام البصري (Visual Fidelity)
- التنفيذ مطابق للتصميم بنسبة 1:1
- احترام:
  - المسافات
  - الأحجام
  - المحاذاة
  - الترتيب البصري

أي اختلاف:
> يُعتبر Bug  
> وليس “تحسين”

---

## 16. قابلية التوسع بدون كسر التصميم
- إضافة Features مستقبلًا:
  - لا تكسر التصميم الحالي
  - لا تتطلب إعادة هيكلة كاملة

التصميم هو الأساس  
والكود يتكيّف معه

---

## 17. الأداء والنظافة التقنية
- لا مكتبات غير ضرورية
- لا CSS غير مستخدم
- لا Logic مكرر
- لا State غير مبرر

كل سطر كود له سبب

---

## 18. التوثيق الخفيف الإلزامي
- Component غير بديهي → تعليق بسيط
- Logic غير مباشر → سبب واضح

بدون:
- شرح زائد
- تعقيد لغوي

---

## 19. الخلاصة (Binding Rule)
> التصميم مُنتهٍ ومُعتمد  
> الكود وظيفته التنفيذ فقط  
> أي قرار بصري خارج التصميم مرفوض
